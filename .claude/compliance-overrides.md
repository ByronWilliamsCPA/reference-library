# Compliance Overrides

Repo-specific overrides for the `repo-compliance` skill. Entries here are skipped
during audit runs without raising FINDING blocks. See
`~/.claude/docs/standards-manifest.yaml` for the canonical check definitions.

| Check ID | Reason | Date added |
| --- | --- | --- |
| TOOL-009 | `basedpyright` is not in qlty's default plugin registry and requires a `pyproject.toml` to function. This is a docs-only repository with no `pyproject.toml`; the two Python scripts in `scripts/` are utility code, not a library. The docs-only type profile already exempts the basedpyright pre-commit hook for the same reason. | 2026-05-14 |
| CI-037 | PR title validation is implemented inline in pr-validation.yml using a conventional-commit regex matching the org's commitlint config. This is functionally equivalent to amannn/action-semantic-pull-request and avoids adding a third-party action dependency for a solo-maintainer docs-only repo. | 2026-05-15 |
| OSSF-006 | Silver badge criteria (two-independent-reviewer code review, bus factor >= 2) are structurally infeasible for a solo-maintainer docs-only repo. Recruiting external contributors solely to satisfy a badge criterion is not a reasonable remediation path. The Passing badge is the appropriate and achievable target. | 2026-05-15 |
| CI-001 | `ci.yml` intentionally does not delegate to the org reusable `python-ci.yml`. This is a docs-only repository; the two utility scripts in `scripts/` are not a library and do not require the full Python CI matrix. The local `ci.yml` verifies required-documentation presence and required-status-checks compatibility, which is the appropriate scope for a docs-only repo. | 2026-05-16 |
| CI-035 | `codeql.yml` sets `cancel-in-progress: false` with an inline rationale because CodeQL scans must complete to produce reliable SARIF for the Security tab. All other PR-triggered workflows use `cancel-in-progress: true` (or the merge-group-aware expression). The deviation is bounded to a single workflow with a documented reason. | 2026-05-16 |
| CI-042 | `publish-unit-test-result-action` is not configured in `ci.yml` because this is a docs-only repository with no test suite and produces no JUnit XML. Adding a no-op annotation step would be cargo-cult. | 2026-05-16 |
| CI-046 | Three publishers outside the default allowlist are accepted as industry-standard for their integrations: `SonarSource/sonarcloud-github-action` (SonarCloud), `anchore/sbom-action` (CycloneDX SBOM generation), `fsfe/reuse-action` (REUSE 3.0 compliance). All three are SHA-pinned per CI-005. | 2026-05-16 |
| OSSF-011 | Documented as an accepted architectural exception in `docs/known-vulnerabilities.md` (entry `SCORECARD:Fuzzing`). Docs-only reference library has no untrusted input surface; Scorecard v5.3.0 does not recognize Python property-based fuzzers like Hypothesis (not installed in this repo; cited as a counter-example of a fuzzer Scorecard fails to detect). Score of 0 accepted. Re-evaluate if a runtime service component is ever added. | 2026-05-16 |
| SCORECARD:publish_results | Documented as an accepted architectural exception in `docs/known-vulnerabilities.md` (entry `SCORECARD:publish_results`). OIDC repository claim in reusable workflows misattributes results to `ByronWilliamsCPA/.github`; `publish-results: false` is forced by the callee. SARIF artifacts remain available in the Actions artifact store. | 2026-05-16 |
| SCORECARD:Fuzzing | Same rationale as OSSF-011; tracked alongside it in `docs/known-vulnerabilities.md`. | 2026-05-16 |
| FOUND-024 | Em-dash cleanup target for repo-authored content (`agents/`, `samples/after/`, `.github/`, `.qlty/`, `.pre-commit-config.yaml`) is tracked inline in `.pre-commit-config.yaml` as `cleanup target 2026-12-31`. No separate issue tracker is in use for a solo-maintainer docs repo; the deadline reminder is via the pre-commit config itself, which is read on every commit. | 2026-05-16 |
| FOUND-025 | This repository uses REUSE 3.0 conventions (`LICENSES/` directory + `REUSE.toml`) instead of a root `LICENSE` file. REUSE compliance is verified in CI via `fsfe/reuse-action`. The global `~/.claude/CLAUDE.md` "LICENSE in every project" baseline predates the REUSE migration; the REUSE pattern carries equivalent licensing intent with per-file SPDX tags and is the chosen convention for ByronWilliamsCPA repos. Note: GitHub's repo homepage license badge and some third-party SBOM tools (FOSSA, Snyk, WhiteSource) may not detect the REUSE layout and report "No license detected." REUSE 3.0 permits a root `LICENSE` file alongside `LICENSES/` for tool compatibility; consider adding a symlink (`LICENSE -> LICENSES/MIT.txt`) if downstream consumers require it. | 2026-05-16 |
